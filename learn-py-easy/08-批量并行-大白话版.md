# 08 - 批量并行：20 个分身同时改代码

## 一句话总结

/batch 是 Claude Code 最强大的技能——把一个大任务拆成 20 个小任务，召唤 20 个分身在独立的代码副本中同时干活，每人创建一个 PR。

---

## 通俗理解

场景：你要把项目中所有 React 组件迁移到 Vue，涉及 500 个文件。

一个人干要一周。但如果你有 20 个助手，每人负责 25 个文件，同时开工，一天就搞定了。

/batch 就是这个"项目管理器"：

```
你: /batch 把所有 React 组件迁移到 Vue

Claude (项目经理):
  "好的，让我先研究一下代码库..."
  (研究 30 秒)

  "我计划拆成 20 个工作单元:
   - 单元 1: src/components/Button → Vue
   - 单元 2: src/components/Form → Vue
   - ...
   - 单元 20: src/pages/Dashboard → Vue

   每个单元独立，互不依赖。
   测试方案: 迁移后跑 npm test。

   确认执行吗？"

你: "确认"

Claude: "开工！召唤 20 个分身..."
  (20 个分身同时开始工作，每人一个独立代码副本)
```

## 三个阶段

### Phase 1: 先想清楚（Plan）

```
┌──────────────────────────────────────────────┐
│  研究阶段                                     │
│                                              │
│  1. 召唤一个研究分身深入分析代码库            │
│  2. 把大任务拆成 5-30 个独立小任务            │
│     ├── 每个任务可以独立完成                  │
│     ├── 每个任务可以独立测试                  │
│     ├── 每个任务可以独立创建 PR               │
│     └── 任务大小尽量均匀                     │
│  3. 确定测试方案                             │
│  4. 写计划文件 → 你审批                      │
│                                              │
│  为什么先规划？                               │
│  → 20 个分身同时改代码，改错了就是灾难        │
│  → 必须确保计划正确才能开始执行               │
└──────────────────────────────────────────────┘
```

### Phase 2: 开干（Execute）

```
┌──────────────────────────────────────────────┐
│  执行阶段                                     │
│                                              │
│  20 个分身同时启动，每个在独立的 Git Worktree  │
│                                              │
│  每个分身的标准流程:                           │
│  1. 改代码 (实现迁移)                         │
│  2. 简化代码 (/simplify)                     │

│  4. 跑端到端测试                              │
│  5. git commit + push + gh pr create         │

│                                              │
│  全部后台运行，不阻塞主流程                    │
└──────────────────────────────────────────────┘
```

## 为什么需要 Git Worktree？

```
没有 Worktree:
  20 个分身在同一个目录改代码 → 互相覆盖 → 一团糟

有 Worktree:
  分身 1: /tmp/worktree-1/ (代码副本 1)
  分身 2: /tmp/worktree-2/ (代码副本 2)
  ...
  分身 20: /tmp/worktree-20/ (代码副本 20)

  每个人改自己的副本，互不干扰
  改完了创建 PR，你来审查合并
```

**类比**：就像 20 个人同时编辑一份 Word 文档会乱套，但每人复印一份各自编辑就没问题。

## 标准化 Worker：统一的 5 步流程

每个分身都遵循相同的流程，就像工厂流水线：

```
Step 1: 实现
  "按照计划修改代码"

Step 2: 简化
  "调用 /simplify 清理代码"

Step 3: 单元测试
  "跑 pytest，失败了就修"

Step 4: 端到端测试
  "按照测试方案验证功能"

Step 5: 提交
  "git commit → push → gh pr create"
  "报告: PR: <url>"
```

统一流程的好处：
- 项目经理（主 Agent）可以统一跟踪
- 结果格式一致，容易解析
- 质量有保证（每步都有检查）

## 安全门控：防止 AI 乱来

```python
register_skill(
    name="batch",
    disable_model_invocation=True,  # Claude 不能自己触发！
    user_invocable=True,             # 只能用户手动输入 /batch
)
```

为什么？因为 /batch 会同时启动 20 个分身改代码，如果 Claude 误触发，后果不堪设想。

就像核弹发射按钮，必须由总统亲自按，不能让 AI 自动按。

## 面试怎么答

**Q: 为什么 /batch 需要 Plan Mode？**
A: 20 个分身同时改代码，一旦开始很难回滚。必须先规划、获得用户审批后再执行，避免大规模错误。

**Q: "PR: \<url\>" 格式为什么重要？**
A: 项目经理需要从每个分身的结果中提取 PR 链接来更新进度表格。统一的格式让解析变得简单可靠。

**Q: 这个模式可以推广吗？**
A: 完全可以。Plan-then-Execute + 隔离执行 + 并行编排 + 标准化 Worker 的模式适用于任何大规模并行任务，不仅限于代码变更。
