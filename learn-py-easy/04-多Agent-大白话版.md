# 04 - 多 Agent：Claude 的分身术

## 一句话总结

当任务太复杂，一个 Claude 搞不定，就召唤"分身"来帮忙。分身有两种：完全独立的"新员工"和共享记忆的"克隆体"。

---

## 通俗理解

你是老板（主 Agent），手下有一堆活要干：

**场景 1**: 你需要有人帮你审查代码
→ 召唤一个**新员工**（Fresh Agent），给他完整的任务说明，让他独立工作
→ 他看不到你之前的对话，但也不会被你的思路影响

**场景 2**: 你需要查一下某个库的用法
→ 召唤一个**克隆体**（Fork Agent），他继承了你所有的记忆
→ 他查完把结果告诉你，中间过程你不用管

**场景 3**: 你要同时改 20 个文件
→ 召唤 20 个**克隆体**，每个人在独立的"工作间"（Git Worktree）里干活
→ 全部并行，互不干扰

---

## 两种分身模式

### Fresh Agent（新员工）

```
┌──────────────┐                    ┌──────────────┐
│  你 (老板)    │                    │  新员工       │
│              │── 给任务说明 ────►  │              │
│  有完整记忆   │                    │  从零开始     │
│  知道上下文   │                    │  需要你告诉他  │
│              │                    │  所有背景     │
│              │◄── 返回结果 ─────  │              │
└──────────────┘                    └──────────────┘

特点:
  ✅ 独立视角，不受你的思路影响
  ❌ 需要完整的任务说明（费 token）
  ❌ 不共享记忆（成本更高）
```

**类比**：就像你把一个任务外包给另一家公司。他们不知道你内部的情况，你需要把所有背景信息都告诉他们。

### Fork Agent（克隆体）

```
┌──────────────┐                    ┌──────────────┐
│  你 (老板)    │                    │  克隆体       │
│              │── 给简短指令 ────►  │              │
│  有完整记忆   │                    │  继承你的记忆  │
│              │◄── 共享记忆 ──────  │              │
│              │                    │  只需知道"做什么"│
│              │◄── 返回结果 ─────  │              │
└──────────────┘                    └──────────────┘

特点:
  ✅ 共享记忆（省 token，成本低）
  ✅ 只需简短指令
  ❌ 不能用不同的模型（会破坏记忆共享）
  ❌ 不要偷看克隆体的中间过程
```

**类比**：就像你让同事帮你查个东西。他知道项目背景，你只需要说"帮我查一下 X 的用法"就行。

---

## Fork 的三条铁律

```
┌──────────────────────────────────────────────────┐
│                                                  │
│  1. 不要偷看 🙈                                   │
│     克隆体会把结果写到文件里，你等通知就行         │
│     不要去读那个文件！会把你搞晕的                │
│                                                  │
│  2. 不要瞎猜 🔮                                   │
│     克隆体还在干活，你不知道结果                  │
│     永远不要编造或预测它的结果                    │
│     用户问"好了吗"，你说"还在跑"，别说"应该快了"  │
│                                                  │
│  3. 指令要简洁 📋                                  │
│     克隆体继承你的记忆，不需要重述背景            │
│     直接说"做什么"就行，像给同事发消息            │
│                                                  │
└──────────────────────────────────────────────────┘
```

## 隔离机制：给每个分身一个独立工位

```
无隔离 (默认):
  分身直接在你的工位上干活
  → 可能不小心改了你正在看的文件
  → 适合: 只读研究任务

Git Worktree 隔离:
  给分身一个独立的工位（代码仓库的副本）
  → 他怎么改都不影响你的代码
  → 改完了告诉你路径，你来审查
  → 没改任何东西？自动清理工位
  → 适合: 并行修改代码

远程隔离:
  在远程服务器上干活
  → 适合: 耗时很长的任务
```

**类比**：
- 无隔离 = 让实习生在你电脑上操作（危险）
- Worktree = 给实习生一台独立的电脑（安全）
- 远程 = 让实习生在家办公（最安全，但通信慢）

---

## 后台执行：不等结果，先干别的

```
前台执行:
  你: "帮我审查这段代码"
  → 等分身审查完 → 拿到结果 → 继续你的工作
  → 适合: 需要结果才能继续的任务

后台执行:
  你: "帮我审查这段代码" (后台)
  → 立刻继续你的工作
  → 分身完成后给你发消息（通知）
  → 你收到通知后再处理结果
  → 适合: 独立任务，可以和其他工作并行
```

**类比**：前台执行像打电话（必须等对方说完），后台执行像发微信（发完就去干别的，对方回了再看）。

## 一张图总结

```
┌──────────────────────────────────────────────────────┐
│              多 Agent 系统                             │
│                                                      │
│  主 Agent (你)                                        │
│  ├── Fresh Agent (新员工)                             │
│  │   └── 独立视角，完整任务说明，不共享记忆           │
│  ├── Fork Agent (克隆体)                              │
│  │   └── 共享记忆，简短指令，低成本                   │
│  ├── 后台 Agent (发微信)                              │
│  │   └── 不等结果，完成后通知                         │
│  └── Worktree Agent (独立工位)                        │
│      └── 在代码副本中工作，互不干扰                   │
│                                                      │
│  核心设计:                                            │
│  ├── Fresh vs Fork: 独立性 vs 成本                    │
│  ├── Worktree: 并行修改代码不冲突                     │
│  ├── 后台执行: 不阻塞主流程                           │
│  └── 通知机制: user-role 消息传递结果                 │
└──────────────────────────────────────────────────────┘
```

## 面试怎么答

**Q: Fork 和 Fresh Agent 的核心区别？**
A: Fork 共享父 Agent 的记忆和 prompt cache（低成本），Fresh Agent 从零开始（高隔离）。Fork 适合研究任务，Fresh 适合需要独立视角的任务（如代码审查）。

**Q: 为什么 Fork 不能用不同的模型？**
A: 不同模型的 prompt cache 格式不同。Fork 的核心优势是共享 cache 省钱，换了模型就失去这个优势了。
